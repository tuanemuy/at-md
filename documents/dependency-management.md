# 依存関係管理

このドキュメントでは、AT-MDプロジェクトにおける依存関係の管理方法と方向性について説明します。

## 依存関係の基本原則

AT-MDプロジェクトでは、以下の基本原則に従って依存関係を管理します：

1. **内側に向かう依存のみ許可**
   - 外側のレイヤーは内側のレイヤーに依存することができる
   - 内側のレイヤーは外側のレイヤーに依存してはならない
   - 依存関係は常に内側に向かう方向のみ

2. **モジュール間の参照は`mod.ts`を経由**
   - モジュール間の参照は必ず`mod.ts`を経由する
   - 他のモジュールのファイルを直接参照してはならない
   - 同一モジュール内のファイルは相対パスで参照する

3. **依存関係の明示的な宣言**
   - 各コンテキストごとに`deps.ts`ファイルを作成し、依存関係を明示的に宣言する
   - モジュール内の実装は`deps.ts`からのre-exportを参照する
   - 外部依存は`deps.ts`を通じて参照する

## レイヤーアーキテクチャ

AT-MDプロジェクトは以下のレイヤーで構成されています：

1. **コアドメイン層（最内層）**
   - ドメインモデル、エンティティ、値オブジェクト、ドメインサービスなど
   - 他のレイヤーに依存しない
   - ビジネスルールとドメインロジックを含む

2. **アプリケーション層**
   - ユースケース、アプリケーションサービス、コマンド、クエリなど
   - コアドメイン層に依存する
   - ビジネスロジックのオーケストレーションを担当

3. **インフラストラクチャ層（最外層）**
   - リポジトリの実装、外部サービスとの連携、データベースアクセスなど
   - アプリケーション層とコアドメイン層に依存する
   - 技術的な詳細を担当

4. **プレゼンテーション層（最外層）**
   - API、UI、コントローラーなど
   - アプリケーション層とコアドメイン層に依存する
   - ユーザーとのインタラクションを担当

## 依存関係の方向性

依存関係の方向性は以下の通りです：

```
プレゼンテーション層 → アプリケーション層 → コアドメイン層
                    ↑
インフラストラクチャ層 ─┘
```

- プレゼンテーション層はアプリケーション層とコアドメイン層に依存する
- インフラストラクチャ層はアプリケーション層とコアドメイン層に依存する
- アプリケーション層はコアドメイン層に依存する
- コアドメイン層は他のレイヤーに依存しない

## コンテキスト間の依存関係

コンテキスト間の依存関係も内側に向かう方向のみ許可されます：

- 各コンテキストは自身のドメインモデルに責任を持つ
- コンテキスト間の連携は、インターフェースを通じて行う
- 共有カーネルを使用して、複数のコンテキストで共有される概念を定義する

## 依存関係の管理方法

### `deps.ts`ファイル

各コンテキストごとに`deps.ts`ファイルを作成し、依存関係を明示的に宣言します：

```typescript
/**
 * コンテキストの依存関係
 * 
 * コンテキストで使用する内部依存をエクスポートします。
 */

// 内部依存
export * from "../other-context/mod.ts";

// 外部依存
export { SomeExternalDependency } from "../../deps.ts";
```

### `mod.ts`ファイル

各コンテキストごとに`mod.ts`ファイルを作成し、外部に公開するAPIを定義します：

```typescript
/**
 * コンテキストの公開API
 * 
 * コンテキストが外部に公開するAPIを定義します。
 */

// エンティティ
export * from "./entities/mod.ts";

// 値オブジェクト
export * from "./value-objects/mod.ts";

// サービス
export * from "./services/mod.ts";
```

## 依存関係の検証

依存関係の検証には以下の方法を使用します：

### 1. コマンドラインでの検証

```sh
deno task check:deps
```

このコマンドは以下をチェックします：

- モジュール間のimportが`mod.ts`を経由しているか
- 他のモジュールのファイルを直接参照していないか
- 依存関係の方向性が正しいか

### 2. リントプラグインによる検証

```sh
deno lint
```

mod-importリントルールが以下をチェックします：

- モジュール間のimportが`mod.ts`を経由しているか
- 違反している場合、修正のヒントを提示

### 3. 循環参照の検出

```sh
deno task check:circular
```

このコマンドは循環参照を検出します：

- モジュール間の循環参照を検出
- ファイル間の循環参照を検出
- 循環参照がある場合、修正のヒントを提示

## ベストプラクティス

### 1. 依存関係の最小化

- 必要最小限の依存関係のみを持つ
- 不要な依存関係は削除する
- 依存関係の数を最小限に抑える

### 2. インターフェースの使用

- 具体的な実装ではなく、インターフェースに依存する
- 依存性の注入を活用する
- テスト容易性を向上させる

### 3. 共有カーネルの活用

- 複数のコンテキストで共有される概念は共有カーネルに定義する
- 共有カーネルは最小限に保つ
- 共有カーネルの変更は慎重に行う

### 4. 依存関係の可視化

- 依存関係を可視化するツールを活用する
- 依存関係のグラフを定期的に確認する
- 問題がある場合は早期に対処する

## 依存関係の例

### コアドメイン層の依存関係

```typescript
// src/core/content/deps.ts
export * from "../common/mod.ts";
export { z } from "../../deps.ts";
export { Result, ok, err } from "../../deps.ts";
```

### アプリケーション層の依存関係

```typescript
// src/application/content/deps.ts
export * from "../../core/content/mod.ts";
export { Result, ok, err } from "../../deps.ts";
```

### インフラストラクチャ層の依存関係

```typescript
// src/infrastructure/content/deps.ts
export * from "../../application/content/mod.ts";
export { Result, ok, err } from "../../deps.ts";
export { eq, and, or, desc, asc, sql, inArray } from "../../deps.ts";
export type { NodePgDatabase } from "../../deps.ts";
```

## 注意点

- 循環参照を避ける
- 依存関係の方向性を常に意識する
- 依存関係の変更は慎重に行う
- 依存関係の検証を定期的に行う
