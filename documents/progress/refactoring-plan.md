# リファクタリング計画と進捗

このドキュメントでは、AT-MDプロジェクトのリファクタリング計画と進捗状況を記録します。

## リファクタリングの目的

以下の項目を改善することを目的としています：

1. テストの粒度を改善
2. モジュール間の依存関係を明示
3. 型の改善
4. オブジェクトのパース処理におけるZodの活用
5. トランザクションの活用
6. その他の改善点

## リファクタリングの進め方

リファクタリングは以下の流れで進めます：

1. **フェーズと作業内容の確認**
   - 各フェーズの目標と作業内容を明確にする
   - 優先順位を決定する

2. **リファクタリング**
   - 計画に基づいて実装を修正する
   - 小さな単位で変更を加える

3. **テスト**
   - 変更した部分のテストを実行する
   - 既存の機能が壊れていないことを確認する

4. **正しくすべてのモジュールに適応できているか確認**
   - 変更が全体に適切に反映されているか確認する
   - 問題があれば2-4を繰り返す

5. **ドキュメントを更新する**
   - 変更内容をドキュメントに反映する
   - 新しい設計パターンや使用方法を説明する

## リファクタリング計画

リファクタリングは以下の3つのフェーズに分けて実施します。

### フェーズ1: 基盤整備（1週間）

1. **Zodの導入と基本スキーマ定義**
   - Zodライブラリの追加
   - 基本的なスキーマ定義の作成
   - ドメインモデルのスキーマ定義

2. **テストヘルパーとファクトリの整備**
   - テストデータファクトリの作成
   - モックヘルパーの作成
   - テストユーティリティの整備

3. **依存関係管理の改善**
   - 各コンテキストごとの`deps.ts`ファイルの作成
   - 依存関係の方向性を明示するドキュメントの作成
   - 循環参照を検出するスクリプトの作成

### フェーズ2: コア改善（2週間）

1. **ドメインモデルの型改善**
   - Zodスキーマを活用した型安全なドメインモデルの実装
   - バリデーション処理の強化
   - 型の厳格化（文字列リテラル型やブランド型の活用）

2. **ユニットテストの追加**
   - ドメインサービスのユニットテスト追加
   - アプリケーションサービスのユニットテスト追加
   - モックを活用した依存関係の分離

3. **トランザクション管理の改善**
   - トランザクション管理の一貫性確保
   - ユニットオブワークパターンの導入検討
   - 複数リポジトリにまたがるトランザクション対応

### フェーズ3: 応用改善（1週間）

1. **パフォーマンス最適化**
   - クエリの最適化
   - キャッシュ戦略の導入
   - N+1問題の解決

2. **ロギング強化**
   - 構造化ロギングの強化
   - ログレベルの適切な使い分け
   - エラーメトリクスの改善

3. **ドキュメント更新**
   - リファクタリングの内容を反映したドキュメントの更新
   - 新しい設計パターンの説明追加
   - コード例の更新

## 次のステップ

フェーズ2（コア改善）が完了しました。次のステップはフェーズ3（応用改善）に進むことです。

### 優先順位

1. **パフォーマンス最適化**
   - クエリ最適化から着手し、特にN+1問題の解決に注力します
   - 次にキャッシュ戦略の検討と実装を行います

2. **ログ強化**
   - 構造化ログの強化を行い、コンテキスト情報を追加します
   - エラーログの監視・アラート連携を整備します

3. **ドキュメント更新**
   - リファクタリング内容を反映したドキュメントを更新します
   - API仕様書を最新の状態に更新します

### 進め方

フェーズ3も小さな単位で進め、各ステップごとにテストを行い、動作確認をします。特にパフォーマンス最適化については、変更前後でのパフォーマンス計測を行い、改善効果を確認します。

## 注意点

- リファクタリングは小さな単位で行い、各ステップでテストを実行する
- 既存の機能を壊さないように注意する
- 変更はドキュメントに反映する
- 依存関係の方向性を常に意識する（内側に向かう依存のみ許可）
