# リファクタリングのテスト戦略

このドキュメントでは、AT-MDプロジェクトのリファクタリングにおけるテスト戦略を記録します。

## テストの目的

リファクタリングにおけるテストの主な目的は以下の通りです：

1. **既存の機能が壊れていないことを確認する**
   - リファクタリングによって意図しない副作用が発生していないか確認する
   - 既存のテストケースが引き続き成功することを確認する

2. **新しい実装が正しく動作することを確認する**
   - 新しく追加された機能や修正された機能が期待通りに動作するか確認する
   - エッジケースや異常系のテストを含める

3. **コードの品質を向上させる**
   - テストカバレッジを向上させる
   - テストコードの可読性と保守性を向上させる

## テスト戦略

### 1. テストの粒度

テストは以下の粒度で実施します：

1. **ユニットテスト**
   - 個々の関数やクラスの振る舞いをテストする
   - 外部依存をモックして分離したテストを行う
   - 境界値や異常系のテストを含める

2. **統合テスト**
   - 複数のコンポーネントの連携をテストする
   - リポジトリとサービスの連携など、レイヤー間の連携をテストする
   - 実際のデータベースを使用したテストを含める

3. **E2Eテスト**
   - エンドツーエンドのユーザーフローをテストする
   - 実際のAPIエンドポイントを使用したテストを行う
   - ユーザーインターフェースの動作をテストする

### 2. テストの実行タイミング

テストは以下のタイミングで実行します：

1. **リファクタリング前**
   - 既存のテストケースを実行し、現在の状態を確認する
   - テストカバレッジを測定し、不足している部分を特定する

2. **リファクタリング中**
   - 小さな変更ごとにテストを実行する
   - 変更した部分に関連するテストを優先的に実行する

3. **リファクタリング後**
   - すべてのテストを実行し、機能が正しく動作することを確認する
   - テストカバレッジを測定し、改善されたことを確認する

### 3. テストヘルパーの活用

テストの効率化と品質向上のために、以下のテストヘルパーを活用します：

1. **テストデータファクトリ**
   - テスト用のデータを簡単に生成するためのファクトリ関数
   - パラメータのオーバーライドによる柔軟なデータ生成

2. **モックヘルパー**
   - 外部依存をモックするためのヘルパー関数
   - リポジトリやサービスのモック作成を簡略化

3. **アサーションヘルパー**
   - 複雑なオブジェクトの検証を簡略化するヘルパー関数
   - 期待値と実際の値の比較を容易にする

## テスト実施手順

### フェーズ1: 基盤整備のテスト

#### 1.1 Zodの導入と基本スキーマ定義のテスト

1. **スキーマの検証テスト**
   - 有効な値がスキーマを通過することを確認
   - 無効な値がスキーマでエラーになることを確認
   - エラーメッセージが適切であることを確認

2. **型の互換性テスト**
   - 既存のコードと新しい型定義の互換性を確認
   - 型の変換が正しく行われることを確認

#### 1.2 テストヘルパーとファクトリの整備のテスト

1. **ファクトリ関数のテスト**
   - デフォルト値が正しく設定されることを確認
   - オーバーライドが正しく適用されることを確認
   - 生成されたオブジェクトが期待通りの構造を持つことを確認

2. **モックヘルパーのテスト**
   - モックが正しく動作することを確認
   - モックのメソッドが期待通りの値を返すことを確認
   - オーバーライドが正しく適用されることを確認

#### 1.3 依存関係管理の改善のテスト

1. **循環参照検出スクリプトのテスト**
   - 既知の循環参照を検出できることを確認
   - 循環参照がない場合に正しく報告することを確認

2. **依存関係の方向性のテスト**
   - 依存関係の方向が正しいことを確認（内側に向かう依存のみ）
   - 不正な依存関係がないことを確認

### フェーズ2: コア改善のテスト

#### 2.1 ドメインモデルの型改善のテスト

1. **バリデーション処理のテスト**
   - 有効な入力が正しく処理されることを確認
   - 無効な入力がエラーになることを確認
   - エラーメッセージが適切であることを確認

2. **型ガードのテスト**
   - 型ガードが正しく動作することを確認
   - 型の絞り込みが正しく行われることを確認

#### 2.2 ユニットテストの追加

1. **ドメインサービスのテスト**
   - 各メソッドが期待通りの結果を返すことを確認
   - エラー処理が正しく行われることを確認
   - 境界値や異常系のテストを含める

2. **アプリケーションサービスのテスト**
   - 外部依存をモックして分離したテストを行う
   - トランザクション処理が正しく行われることを確認
   - エラー処理が正しく行われることを確認

#### 2.3 トランザクション管理の改善のテスト

1. **トランザクションの一貫性テスト**
   - トランザクションが正しくコミットされることを確認
   - エラー時にロールバックされることを確認
   - 複数のリポジトリにまたがるトランザクションが正しく動作することを確認

2. **ユニットオブワークのテスト**
   - ユニットオブワークパターンが正しく動作することを確認
   - 複数の操作が一つのトランザクションで実行されることを確認

## テスト自動化

テストの自動化のために、以下のスクリプトを用意します：

1. **ユニットテスト実行スクリプト**

```bash
deno task test:unit
```

2. **統合テスト実行スクリプト**

```bash
deno task test:integration
```

3. **E2Eテスト実行スクリプト**

```bash
deno task test:e2e
```

4. **すべてのテスト実行スクリプト**

```bash
deno task test
```

5. **カバレッジ測定スクリプト**

```bash
deno task test:cov
```

これらのスクリプトは`deno.json`に定義し、CIパイプラインでも実行できるようにします。
