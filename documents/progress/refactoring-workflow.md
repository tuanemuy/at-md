# リファクタリングのワークフロー

このドキュメントでは、AT-MDプロジェクトのリファクタリングにおけるワークフローを記録します。

## リファクタリングの基本的な流れ

リファクタリングは以下の流れで進めます：

1. **フェーズと作業内容の確認**
   - 各フェーズの目標と作業内容を明確にする
   - 優先順位を決定する
   - 依存関係を考慮して作業順序を決める

2. **リファクタリング**
   - 計画に基づいて実装を修正する
   - 小さな単位で変更を加える
   - コミットメッセージに変更内容を明記する

3. **テスト**
   - 変更した部分のテストを実行する
   - 既存の機能が壊れていないことを確認する
   - テストカバレッジを測定する

4. **正しくすべてのモジュールに適応できているか確認**
   - 変更が全体に適切に反映されているか確認する
   - 問題があれば2-4を繰り返す
   - 依存関係の方向性が正しいことを確認する

5. **ドキュメントを更新する**
   - 変更内容をドキュメントに反映する
   - 新しい設計パターンや使用方法を説明する
   - コード例を更新する

## 各ステップの詳細

### 1. フェーズと作業内容の確認

#### 作業内容

1. **リファクタリングの目的を明確にする**
   - 何を改善するのか
   - なぜ改善が必要なのか
   - どのような効果が期待できるのか

2. **作業範囲を決定する**
   - どのモジュールを対象とするのか
   - どのレイヤーを対象とするのか
   - どのファイルを対象とするのか

3. **優先順位を決定する**
   - 依存関係の少ないモジュールから始める
   - 影響範囲の小さい変更から始める
   - 基盤となる部分から始める

#### 成果物

- リファクタリング計画書
- 作業項目リスト
- スケジュール

### 2. リファクタリング

#### 作業内容

1. **コードの分析**
   - 現在のコードの構造を理解する
   - 問題点を特定する
   - 改善方法を検討する

2. **変更の実装**
   - 小さな単位で変更を加える
   - 一度に一つの問題に対処する
   - コードの可読性と保守性を向上させる

3. **コミット**
   - 変更内容を明記したコミットメッセージを書く
   - 関連するIssueやPull Requestを参照する
   - 変更の理由を説明する

#### 成果物

- 修正されたコード
- コミット履歴
- 変更内容の説明

### 3. テスト

#### 作業内容

1. **ユニットテストの実行**
   - 変更した部分のユニットテストを実行する
   - 新しい機能のテストを追加する
   - エッジケースのテストを追加する

2. **統合テストの実行**
   - 変更した部分の統合テストを実行する
   - 他のモジュールとの連携をテストする
   - 実際のデータベースを使用したテストを行う

3. **E2Eテストの実行**
   - エンドツーエンドのユーザーフローをテストする
   - 実際のAPIエンドポイントを使用したテストを行う
   - ユーザーインターフェースの動作をテストする

#### 成果物

- テスト結果
- テストカバレッジレポート
- 問題点と改善点のリスト

### 4. 正しくすべてのモジュールに適応できているか確認

#### 作業内容

1. **依存関係の確認**
   - 依存関係の方向性が正しいことを確認する
   - 循環参照がないことを確認する
   - 不要な依存関係がないことを確認する

2. **型の互換性の確認**
   - 型の変更が他のモジュールに影響していないか確認する
   - 型の変換が正しく行われていることを確認する
   - 型エラーがないことを確認する

3. **パフォーマンスの確認**
   - 変更がパフォーマンスに影響していないか確認する
   - ボトルネックがないか確認する
   - メモリ使用量が適切であることを確認する

#### 成果物

- 依存関係の分析結果
- 型チェックの結果
- パフォーマンス測定結果

### 5. ドキュメントを更新する

#### 作業内容

1. **コードドキュメントの更新**
   - コメントを更新する
   - 型定義を更新する
   - 関数のシグネチャを更新する

2. **設計ドキュメントの更新**
   - アーキテクチャ図を更新する
   - クラス図を更新する
   - シーケンス図を更新する

3. **ユーザードキュメントの更新**
   - APIドキュメントを更新する
   - 使用方法を更新する
   - コード例を更新する

#### 成果物

- 更新されたドキュメント
- 変更履歴
- 新しい設計パターンの説明

## リファクタリングの進め方

リファクタリングは以下の方針で進めます：

1. **小さな単位で変更する**
   - 一度に一つの問題に対処する
   - 変更の影響範囲を最小限に抑える
   - 頻繁にコミットする

2. **テストを重視する**
   - 変更前にテストを実行する
   - 変更後にテストを実行する
   - テストカバレッジを向上させる

3. **ドキュメントを更新する**
   - 変更内容をドキュメントに反映する
   - 新しい設計パターンや使用方法を説明する
   - コード例を更新する

4. **レビューを受ける**
   - 変更内容をレビューしてもらう
   - フィードバックを反映する
   - 問題点を修正する

## リファクタリングの記録

リファクタリングの進捗は以下の形式で記録します：

```
# リファクタリング記録: <リファクタリング名>

## 実施日時
YYYY-MM-DD HH:MM:SS

## 対象
- モジュール: <モジュール名>
- ファイル: <ファイル名>
- 関数/クラス: <関数/クラス名>

## 変更内容
- <変更内容1>
- <変更内容2>
...

## テスト結果
- 成功: <成功したテスト数>
- 失敗: <失敗したテスト数>
- スキップ: <スキップしたテスト数>

## カバレッジ
- ライン: <ラインカバレッジ>%
- ブランチ: <ブランチカバレッジ>%

## 問題点
- <問題点1>
- <問題点2>
...

## 改善点
- <改善点1>
- <改善点2>
...
```

## リファクタリングのチェックリスト

リファクタリングが完了したら、以下のチェックリストを確認します：

- [ ] すべてのテストが成功している
- [ ] テストカバレッジが向上している
- [ ] 依存関係の方向性が正しい
- [ ] 循環参照がない
- [ ] 型エラーがない
- [ ] パフォーマンスが向上している（または少なくとも低下していない）
- [ ] ドキュメントが更新されている
- [ ] コードの可読性と保守性が向上している
- [ ] 変更内容がレビューされている
- [ ] フィードバックが反映されている
